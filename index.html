Vehicle Inventory Management - A Technical Design Document

Backend solution to support vehicle inventory maintenance.

View on GitHub

Overview

We will create a vehicle inventory management system to support front-end clients' needs which include the following use-cases :

  1. Dealership web site support for viewing data related to vehicle inventory via an existing CMS application.
  2. Updating inventory data via an existing admin app.

Assumptions

Environment We have production and non-production aws accounts with full access in the non-prod accounts to create, modify, and deliver our solution.

Team Experience We have a team of developers that are familiar with

  1. fluent in cicd tooling (terraform, codebuild, roles and policies, proper tagging, and Cloudwatch monitoring/alerting.
  2. fluent in the following languages: Python and NodeJS

Data

  1. Each payload (file) represents ALL the inventory for an individual dealer.

  2. Dealers are allowed only one feed per 24 hours period.

  3. The dealer id is embedded in the filename, adhering to pre-defined naming conventions.

  4. Dealers have been vetted and are managed in separate systems. Access to dealer contact info is available to our solution.

  5. If a dealer id is not valid, the system will discard the feed for that "unknown" dealer.

  6. Industry standards exist to define appropriate values for vehicle properties based on OEM build specs.

  7. Inventory feeds are received nightly in a csv format.

  8. The processing and syndication of feeds does not take longer than a 24 hour window.

  9. Feeds taking longer then the predefined window shall be treated as stale and appropriate notifications shall be sent.

  10. The processing and syndication of feeds will not negatively effect client application performance.

  11. The order in which payloads are received is not significant.

  12. Redundant feeds do not cause adverse effects to overall inventory. (updating inventory should be idempotent)

  13. Sourced inventory data should be scrubbed and validated, setting aside invalid payloads and notifying the authors of such payloads. a. Validation Rules (all fields are REQUIRED) i. Year -> 4 digit valid year (Years may include future) ii. Make -> industry standard values and abbreviations iii. Model -> industry standard values and abbreviations iv. BodyStyle -> industry standard values and abbreviations v. vin -> must be a valid vin (leverage existing vin validation routines) vi. color -> as defined by the oem vii. mileage -> round up to whole number viii. msrp -> with 2 digits after decimal point ix. asking_price -> with 2 digits after decimal point

Inventory Management API Client applications will access inventory data via a published api leveraging GraphQL. Why GraphQL over REST?
GraphQL appears to best-fit for the existing client applications and use-cases. Clients retrieve exactly what they need and no more. This is done through pre-defined contracts depicting all the available data the client will have access to.
In addition, the number of requests is mitigated because the backend will pull from any number of tables to build and tailor the response.

Audience and Traffic Peak hours for client activity occurs during standard business hours (8am to 7pm EST). Sourced inventory data should be validated, setting aside invalid payloads and notifying the authors of such payloads. This requires "saving" the invalid payloads "as-is" for further investigative actions.
